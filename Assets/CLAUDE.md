# CLAUDE.md / AGENTS.md

**절충점:**
이 지침은 속도보다 신중함을 우선하도록 유도한다.
아주 사소한 작업에서는 상황에 맞게 판단한다.

## 1. 코딩하기 전에 먼저 생각하기

**추측하지 않는다. 혼란을 숨기지 않는다. 선택지와 절충점을 드러낸다.**

구현하기 전에:
- 자신의 가정을 명확하게 말한다. 확실하지 않다면 질문한다.
- 여러 해석이 가능하다면, 조용히 하나를 고르지 않고 가능한 해석들을 제시한다.
- 더 단순한 접근이 있다면 말한다. 필요하다면 사용자의 요청에도 이견을 제시한다.
- 불명확한 부분이 있다면 멈춘다. 무엇이 혼란스러운지 이름 붙여 설명하고 질문한다.

## 2. 단순함을 먼저 선택하기

**문제를 해결하는 데 필요한 최소한의 코드만 작성한다. 추측성 코드는 작성하지 않는다.**

- 요청받은 범위를 넘어서는 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 만들지 않는다.
- 요청받지 않은 '유연성'이나 '설정 가능성'을 추가하지 않는다.
- 실제로 발생할 수 없는 상황을 위한 오류 처리를 추가하지 않는다.
- 200줄로 작성했는데 50줄로도 충분하다면, 다시 단순하게 작성한다.

스스로에게 묻는다: '숙련된 시니어 엔지니어가 이 코드를 과하게 복잡하다고 볼까?' 그렇다면 단순화한다.

## 3. 필요한 부분만 정확히 수정하기

**반드시 필요한 부분만 건드린다. 자신이 만든 문제만 정리한다.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포매팅을 '개선'하려고 하지 않는다.
- 고장 나지 않은 코드를 리팩터링하지 않는다.
- 자신이 다르게 작성하고 싶더라도, 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견했다면 언급만 한다. 삭제하지 않는다.

자신의 변경으로 고아 코드가 생겼다면:
- 자신의 변경 때문에 더 이상 사용되지 않게 된 import, 변수, 함수는 제거한다.
- 원래부터 존재하던 죽은 코드는 요청받지 않는 한 제거하지 않는다.

검증 기준: 변경된 모든 줄은 사용자의 요청과 직접 연결되어 있어야 한다.

## 4. 목표 중심으로 실행하기

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 바꾼다:
- '검증 로직 추가' → '잘못된 입력에 대한 테스트를 작성한 뒤, 그 테스트를 통과시키기'
- '버그 수정' → '버그를 재현하는 테스트를 작성한 뒤, 그 테스트를 통과시키기'
- 'X 리팩터링' → '리팩터링 전후 모두 테스트가 통과하는지 확인하기'

여러 단계가 있는 작업에서는 짧은 계획을 먼저 말한다:
```text
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

성공 기준이 명확하면 독립적으로 반복하며 작업할 수 있다.
성공 기준이 약하면, 예를 들어 '작동하게 만들기'처럼 모호하면, 계속해서 추가 확인이 필요하다.

## 5. 코드 스타일 규칙 (이 프로젝트 강제)

**아래 규칙은 예외 없이 적용한다. 어기면 사용자가 롤백 요청한다.**

### 5-1. 주석
- **언어**: 한국어
- **금지**: 화살표/아이콘/전각기호 (예: `→`, `✓`, `★`). 'A -> B' 대신 'A에서 B로 전환'
- **간결성**: 함수 위 1줄, 변수 옆 짧은 단어, 위험 분기 1줄 경고만
- **파일 헤더 블록**은 보존 (제거 금지, 새 파일도 기존 스타일 따라 작성)

### 5-2. C# 문법
- **`var` 금지** → 항상 명시적 타입
  - O: `int x = 0;` / `string s = "";`
  - X: `var x = 0;`
- **`#region` / `#endregion` 금지** (발견 시 제거)

### 5-3. Bool 비교
- O: `if (isReady == false)`
- X: `if (!isReady)`

### 5-4. 중괄호 / 줄 단위
- **Allman 스타일**: `{` 는 다음 줄
  ```csharp
  if (cond)
  {
      doSomething();
  }
  ```
  X: `if (cond) {`
- **한 줄 한 선언, 한 줄 한 문장**
  - O:
    ```csharp
    int a = 0;
    int b = 0;
    ```
  - X: `int a = 0, b = 0;` / `a = b = 0;` / `a = 0; b = 0;`

## 6. Memory.md 자동 갱신

**작업 진행 상황을 Memory.md 에 자동 기록한다.**

다음 상황이 발생할 때마다 Memory.md 를 갱신한 파일을 출력한다:
- 스크립트의 수정/리팩터링이 완료되어 더 이상 건드릴 필요가 없을 때
- 새 스크립트가 작성되어 완성되었을 때
- 씬 구성, ScriptableObject 에셋, 프리팹 등 유니티 작업이 완료되었을 때
- 일괄 작업(예: 입력 시스템 교체, 코딩 스타일 적용 등)이 완료되었을 때

갱신 규칙:
- 사용자가 명시적으로 요청하지 않아도 자동으로 갱신한다.
- 갱신 시 기존 항목을 삭제하지 않고 추가만 한다.
- 한 줄로 끝나는 항목보다 카테고리별로 정리한다.
- 작업 단위가 작은 경우(1~2줄 변경)는 굳이 기록하지 않는다.
- 새 대화 시작 시 사용자가 Memory.md 를 첨부하면 그 시점부터 이어서 작업한다.
